# 星辰引路者当前已实现功能介绍

本文档用于说明项目当前已经实现并接入主链路的功能、运行方式以及当前边界，方便开发、联调和演示时快速了解系统现状。

## 1. 项目当前整体能力

当前系统由两部分组成：

- 硬件端：运行在 ESP32-S3 上，负责环境感知、语音录制与播报、GPS 数据采集与上传
- 服务端：运行在 `web/app.py`，负责大模型意图识别、导航计算、天气查询、GPS 位置缓存与反解

目前已经稳定接入主流程的能力有：

- 语音对话
- 语音导航
- 天气查询
- GPS 定位与位置上传
- 超声波避障提醒
- 光敏检测与照明控制
- 按钮触发语音交互

当前未接入主链路或已移除的能力有：

- 视觉识别相关功能（盲道、红绿灯、摄像头推理）
- 服务端语音合成接口
- 服务端 ASR 接口
- 阿里云语音合成能力

## 2. 硬件端当前功能

### 2.1 语音交互

硬件端支持两种方式进入语音交互：

- 唤醒词触发
- 按钮触发

交互流程如下：

1. 播放提示音
2. 打开麦克风录音
3. 本地结束录音
4. 调用百度语音识别获取文字
5. 将识别结果发送给服务端 `/ai`
6. 根据服务端返回内容进行播报

当前语音交互已经改为**单次对话模式**：

- 每次触发只录音一次
- 回答播报完成后直接回到待唤醒状态
- 不再在回答后自动继续打开麦克风等待下一句

### 2.2 百度语音服务

硬件端当前仍使用百度语音能力：

- 百度语音识别：语音转文字
- 百度语音合成：文字转语音播报

系统启动时会主动申请百度 `access_token`。  
为了适配网络波动，当前已经加入：

- 更长的 HTTP 超时时间
- DNS 解析重试
- 显式 DNS 配置

### 2.3 GPS 定位

硬件端已经接入 GPS 模块主链路。

当前实现方式：

- 使用 `UART1` 读取 GPS 模块输出的 NMEA 数据
- 解析 RMC 语句获取经纬度
- 周期性将最新有效坐标上传到服务端 `/gps`

当前接入的 GPS 模块信息：

- 芯片：中科微电子 ATGM332D
- 类型：GPS + BDS 集成模块

### 2.4 超声波避障

硬件端保留并启用了超声波避障能力：

- 使用超声波模块检测前方障碍物
- 当距离低于阈值时，触发振动和蜂鸣器提醒

该部分当前仍是本地实时运行，不依赖服务端。

### 2.5 光敏与照明

硬件端保留并启用了光敏检测能力：

- 读取光敏模块数字输出
- 在环境变暗时自动打开照明
- 环境恢复明亮时自动关闭照明

### 2.6 按钮触发

测试/语音按钮仍然有效：

- 按下按钮可直接触发一轮语音交互

## 3. 服务端当前功能

服务端主程序位于：

- [app.py](D:\Intelligent-Guide-Cane\web\app.py)

当前保留的核心接口有：

- `/ai`
- `/gps`
- `/weather`
- `/navigation_update`
- `/exit_navigation`

### 3.1 `/ai`：大模型意图识别与业务分流

`/ai` 是当前语音主链路的核心入口。

硬件端把语音识别出的文本发送到 `/ai` 后，服务端会调用千问进行意图识别，并分流为以下几类：

- 导航请求
- 退出导航
- 天气查询
- 普通对话

服务端当前保留了千问能力，用于：

- 判断用户意图
- 生成普通对话回复

### 3.2 天气查询

天气查询功能已经接回主链路。

当前实现方式：

- 服务端使用和风天气 API 查询城市天气
- 用户说出天气问题后，由千问输出结构化天气意图
- 服务端根据意图查询天气并返回简洁中文结果

当前天气默认策略：

- 若 GPS 已上传并成功反解城市，则优先使用当前位置所在城市
- 若没有当前位置，则使用默认城市

### 3.3 导航能力

导航功能已经接入主链路。

当前实现方式：

- 用户发起导航请求后，服务端通过高德地图接口获取目的地坐标
- 使用高德步行导航接口计算路线
- 返回第一步导航指令给硬件端播报
- 导航开始后，硬件端定时调用 `/navigation_update`
- 服务端结合最新 GPS 位置重新计算导航状态，并返回下一步指令

当前导航起点策略：

- 优先使用最近上传的 GPS 经纬度作为起点
- 如果没有有效 GPS，则退回默认位置

### 3.4 GPS 接收与城市反解

服务端当前已经支持接收硬件端 GPS 坐标：

- `/gps` 接收经纬度
- 收到坐标后调用高德逆地理编码
- 缓存当前设备对应的：
  - 经度
  - 纬度
  - 当前城市
  - 区县
  - 详细地址

这些信息会被后续能力复用：

- 天气查询默认城市
- 导航默认起点

## 4. 当前接口关系

硬件端与服务端当前主链路如下：

### 4.1 对话链路

1. 用户说话
2. ESP32 录音
3. 百度语音识别得到文字
4. 发送到服务端 `/ai`
5. 千问识别意图并生成响应
6. 返回 JSON
7. ESP32 解析 `response`
8. 百度语音合成播报

### 4.2 GPS 链路

1. GPS 模块输出 NMEA
2. ESP32 解析有效经纬度
3. 周期上传到服务端 `/gps`
4. 服务端反解当前城市与地址
5. 天气与导航流程复用当前位置

### 4.3 导航链路

1. 用户发出导航请求
2. `/ai` 识别为导航意图
3. 服务端调用高德规划路线
4. ESP32 播报第一步指令
5. ESP32 周期请求 `/navigation_update`
6. 服务端根据 GPS 位置更新导航步骤
7. ESP32 播报下一步指令

## 5. 当前使用到的外部服务

当前项目已经接入并在主流程使用的外部服务有：

- 百度语音识别
- 百度语音合成
- 阿里云 DashScope / 千问
- 和风天气
- 高德地图（地理编码、逆地理编码、步行导航）

当前未在主链路使用的服务有：

- 阿里云语音合成
- 服务端本地/云端 TTS 接口

## 6. 当前限制与说明

### 6.1 GPS 依赖环境

GPS 是否能正常拿到定位，强依赖：

- 天线状态正常
- 供电稳定
- 室外开阔环境

如果 GPS 只有 NMEA 输出但始终无有效坐标，则天气和导航会回退到默认位置/默认城市。

### 6.2 网络依赖

语音与服务端能力都依赖 WiFi 外网连接。

尤其在启动阶段：

- 百度 `access_token` 获取容易受 DNS 和外网环境影响
- 已加入重试和超时机制，但网络质量仍会影响体验

### 6.3 单次对话模式

当前语音链路已经关闭连续对话：

- 每次触发只处理一轮
- 回答完成后不会自动继续监听下一句

这更适合当前稳定性优先的使用方式。

## 7. 当前适合演示的能力

基于目前实现状态，适合演示的功能有：

- 唤醒后问答
- 导航到指定地点
- 查询当前城市或指定城市天气
- GPS 坐标上传与服务端识别当前城市
- 超声波本地避障提醒
- 光敏自动照明

## 8. 建议的后续方向

如果继续迭代，建议优先关注以下方向：

- 提升 GPS 定位稳定性与冷启动速度
- 优化百度 Token 获取时的网络容错
- 降低超声波错误日志刷屏频率
- 在服务端增加设备标识，而不仅仅依赖 IP 区分设备
- 恢复视觉能力时，单独走独立模块，不直接污染现有语音/GPS 主链
